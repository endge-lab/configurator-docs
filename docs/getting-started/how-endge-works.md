# Как работает Endge

Endge превращает описание системы в исполняемую программу. Разработчик и аналитик описывают предметную область, правила, сценарии и представления приложения в source-модели, а платформа проверяет это описание, собирает программу и передаёт её в нужное runtime-окружение.

Главная идея Endge состоит в том, что между замыслом системы и её техническим исполнением существует явный, управляемый путь.

```text
Бизнес-задача
   ↓
Source-модель в DSL или конфигураторе
   ↓
Базовая модель и контекстные переопределения
   ↓
Компиляция и проверка
   ↓
Program
   ↓
Runtime
   ↓
Frontend, backend и интеграции
   ↓
Диагностика и дальнейшее развитие в IDE
```

## Контекст: для кого и где работает система

Контекст определяет границы, в которых Endge собирает и исполняет приложение. Он отвечает не только на вопрос «какой код нужно запустить», но и на вопросы: для какой организации, какого проекта и какого окружения собирается конкретная версия системы.

В типовом случае контекст складывается из нескольких уровней:

- **Workspace** — пространство работы команды: набор участников, общих ресурсов и связанных проектов.
- **Tenant** — граница отдельной организации, заказчика или изолированного клиента системы.
- **Project** — конкретная прикладная модель, которую команда разрабатывает и поставляет.
- **Environment** — среда, в которой программа будет исполняться: локальная разработка, тестирование, staging или production.

Эти уровни не обязательно образуют одну жёсткую вложенную иерархию. Например, один workspace может содержать несколько проектов, один project может поставляться нескольким tenant, а для каждого из них может существовать несколько environment. Вместе они определяют, какая исходная модель, какие переопределения, данные и настройки относятся к текущему запуску.

## Source: исходное описание системы

Работа с Endge начинается не с создания отдельных экранов, API-методов или таблиц базы данных. Сначала команда описывает саму систему: какие сущности в ней существуют, как они связаны, какие правила действуют и какие сценарии должны быть доступны пользователю.

Это описание называется **source**.

Source хранит устойчивую модель приложения:

- сущности и их поля;
- связи между сущностями;
- действия, события и правила;
- запросы и преобразования данных;
- представления и компоненты;
- политики доступа;
- конфигурацию отдельных runtime-модулей.

Source можно редактировать двумя взаимодополняющими способами:

- через DSL (Domain Specific Language) — когда требуется точное текстовое описание, контроль изменений и работа со сложными конструкциями;
- через визуальный конфигуратор — когда нужно проектировать сущности, связи и структуру приложения без постоянного обращения к исходному коду.

Это не две разные системы и не два независимых способа разработки. DSL и конфигуратор работают с одной моделью, но подходят для разных задач и ролей в команде.

## Базовая модель и контекстные переопределения

Обычно приложение не существует только в одном варианте. Оно может поставляться разным заказчикам, работать в разных странах, иметь разные роли пользователей или требовать отдельных интеграций.

В Endge сначала создаётся базовая модель. Она содержит общие сущности, процессы и правила, которые должны быть одинаковыми для всех вариантов системы.

Затем к ней может быть применён конкретный контекст. Контекст не создаёт отдельную копию проекта, а точечно уточняет базовую модель с помощью **Specific Override Pattern**.

Например, в базовой системе существует сущность «Задача»:

```text
Задача
├── название
├── исполнитель
├── срок
└── статус
```

Для одного заказчика можно добавить обязательное согласование руководителем:

```text
Контекст заказчика
├── поле «Согласовано»
└── правило: задачу нельзя завершить без подтверждения
```

Базовая сущность при этом не переписывается. Endge объединяет общий каркас с переопределениями выбранного контекста и получает итоговую модель для конкретной поставки.

Это позволяет развивать общую систему без копирования репозитория и без переноса одинаковых изменений между несколькими версиями приложения.

::: warning Важно: контекст поставки
При компиляции Endge собирает программу для выбранного контекста. В итоговую поставку попадают только применимые сущности, правила, переопределения и конфигурация конкретного workspace, tenant, project и environment. Заказчик получает предназначенную для него версию системы, а не весь набор вариантов, существующих в общей модели.

Компиляция по контексту уменьшает объём поставки и исключает ненужные части модели, но сама по себе не заменяет контроль доступа, изоляцию данных и политики безопасности. Эти границы должны обеспечиваться также runtime, хранилищем и механизмами аутентификации.
:::

## Компиляция: от source к program

Source — это авторское описание системы. Оно удобно для проектирования, редактирования и анализа, но не обязательно должно напрямую исполняться в runtime.

Перед запуском Endge передаёт source в компилятор. Компилятор:

- проверяет, что ссылки между сущностями корректны;
- проверяет совместимость типов и контрактов;
- применяет выбранные контекстные переопределения;
- собирает итоговую структуру приложения;
- формирует производные данные, нужные для исполнения;
- создаёт **program** — подготовленную программу для runtime.

Важно различать эти два состояния:

- **Source** — то, что создаёт и хранит команда.
- **Program** — результат компиляции, подготовленный для исполнения.

Это разделение позволяет сделать ошибки понятнее, а runtime — более предсказуемым. Команда работает с выразительной исходной моделью, а среда исполнения получает уже проверенную и собранную программу.

::: info Headless-выполнение
Скомпилированная program не зависит от визуализации: ей не требуется браузер, DOM или renderer, чтобы участвовать в выполнении системы. Она может быть передана в headless runtime — среду исполнения без пользовательского интерфейса — и использоваться для тестов, фоновых задач, серверных процессов, интеграций и удалённых сервисов.

При этом program не является хранилищем живого состояния. Она задаёт скомпилированную структуру и поведение, а данные, реактивность и жизненный цикл конкретного запуска принадлежат экземпляру runtime.
:::

## Runtime: когда программа становится живой

После компиляции начинает работать runtime.

Program описывает, что система умеет: какие сущности, правила, действия и представления в ней существуют. Runtime отвечает за то, чтобы это работало в конкретный момент времени — с реальными данными, текущим пользователем, состоянием приложения и внешними событиями.

Runtime может:

- загрузить и обновить данные;
- создать живое представление сущности;
- обработать действие пользователя;
- выполнить команду;
- передать событие в интеграцию;
- обновить зависимые части интерфейса;
- управлять жизненным циклом ресурсов и компонентов.

Например, пользователь нажимает кнопку завершения задачи. Runtime определяет, доступно ли это действие в текущем контексте, проверяет правило согласования, выполняет команду, обновляет состояние задачи и передаёт изменения в связанные представления.

## Исполнение в конкретной технологии

Модель Endge не привязана напрямую к одному способу отображения или одному backend-фреймворку.

Она описывает, какое поведение и представление нужны системе. А конкретный adapter или runtime определяет, как это будет исполнено в выбранной технологии.

На frontend программа может быть отображена, например, через:

- Vue и DOM;
- Nova и Canvas;
- другой поддерживаемый renderer.

На backend одна и та же предметная модель может использоваться для построения:

- прикладных сервисов;
- обработчиков команд;
- правил валидации;
- API;
- интеграционных процессов;
- механизмов хранения данных.

Это не означает, что любую модель можно автоматически запустить в любой технологии. Для каждой целевой среды должен существовать соответствующий compiler path, runtime или adapter.

::: warning Важный принцип
Предметная модель не должна переписываться только из-за смены технического способа её исполнения.
:::

## IDE и наблюдаемость

IDE Endge нужна не только для редактирования source. Она должна помогать команде видеть путь системы целиком:

```text
Source
→ применённые переопределения
→ ошибки и результат компиляции
→ Program
→ Runtime-состояние
→ события, команды и отображение
```

Это позволяет понять:

- какая исходная сущность породила конкретное поведение;
- какое переопределение сработало в текущем контексте;
- почему компилятор не принял модель;
- какое состояние находится в runtime;
- откуда пришло событие и что оно изменило;
- где требуется изменение модели, а где — императивное расширение.

Таким образом, IDE становится инструментом не только для редактирования, но и для диагностики и управления всем pipeline разработки приложения.

## Императивные расширения

<div class="endge-principle endge-principle--blue" role="note">
  <div class="endge-principle__title" aria-hidden="true">!</div>
  <p>Endge не пытается заменить конфигурацией весь код приложения.</p>
</div>

Императивный код остаётся необходимым для сложных вычислений, нестандартных алгоритмов, внешних интеграций, специализированных renderer-ов и инфраструктурных адаптеров.

Разница в том, что такой код не должен размывать основную модель системы. Он подключается в определённых точках расширения и дополняет декларативный каркас.

> Декларативная модель отвечает за устойчивую структуру системы, а императивный код — за уникальное поведение, которое нельзя или не нужно выражать только конфигурацией.
