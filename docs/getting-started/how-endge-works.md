# Как работает Endge

Пользователь создаёт приложение как набор связанных документов. Каждый документ имеет стабильный идентификатор, тип и исходное описание. Конфигуратор сохраняет эти документы в домене, но сам не является их runtime.

## Основной поток

1. Пользователь открывает проект и окружение.
2. Создаёт или изменяет документ в конфигураторе.
3. Документ сохраняется как source.
4. Компилятор проверяет source и строит программу.
5. Runtime создаёт исполняемый host с собственным жизненным циклом.
6. Результат попадает в данные, события или слой представления.

## Три разных состояния

| Состояние | Что содержит | Где используется |
| --- | --- | --- |
| Source | Авторское описание | хранение и редактирование |
| Program | проверенный исполняемый контракт | граница компилятора и runtime |
| Runtime | живое состояние конкретного запуска | выполнение, реактивность, события |

Изменение runtime не должно незаметно переписывать source. Обратное сохранение — отдельная пользовательская операция.

## Как документы связываются

Документы ссылаются друг на друга через identity. Например, Query получает данные, DataView преобразует их, Computation вычисляет производное значение, а ComponentSFC показывает результат. Связи собираются в композицию или задаются через порты компонента.

## Где находится конфигуратор

Конфигуратор управляет авторским процессом: показывает дерево домена, открывает редакторы, запускает preview и диагностику. Он использует те же контракты ядра, но не подменяет собой compiler или runtime.

Детальная схема границ приведена в разделе [Source → Compiler → Program → Runtime](/architecture/source-compiler-program-runtime).
