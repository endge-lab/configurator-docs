# Диагностика и отладка

Диагностика отвечает на вопрос «что произошло и в каком контексте», а debugger помогает исследовать текущее состояние конкретного запуска.

## Что фиксировать

- идентификатор операции и correlation id;
- тип и identity доменного документа;
- этап `compile`, `create`, `execute` или `dispose`;
- длительность и итог операции;
- безопасные метаданные ошибки;
- связи с родительским trace или событием.

## Уровни проверки

| Уровень | Пример проблемы | Инструмент |
| --- | --- | --- |
| Source | синтаксическая ошибка | diagnostics редактора |
| Compiler | несовместимый контракт | compiler diagnostic |
| Program | неверная или устаревшая программа | program inspector |
| Runtime | host не создан или завершился ошибкой | runtime debugger |
| Data | нет входа или output | Raph/store inspector |
| Presentation | адаптер не отобразил готовое состояние | UI adapter tools |

## Безопасность

В журнал не должны попадать токены, пароли, полные credential-объекты и чувствительные payload. Диагностическая запись должна сохранять достаточно контекста для поиска причины, но не становиться копией пользовательских данных.

## Рекомендуемый порядок разбора

1. Найдите первую ошибку операции, а не последний побочный warning.
2. Проверьте identity и версию source.
3. Свяжите compiler diagnostic с program artifact.
4. Проверьте lifecycle runtime-host.
5. Затем исследуйте входы, outputs, события и presentation.

Такой порядок уменьшает количество ложных гипотез при отладке.
